# ADR 003. Star Schema 기반 Data Mart 설계

- **날짜**: 2025.08
- **상태**: 승인됨

---

## 배경

Silver Layer의 정제된 데이터를 분석 레이어(Gold)로 변환할 때 데이터 구조를 결정해야 했습니다.
주요 후보는 두 가지였습니다.

- **Option A**: Silver Layer 구조 그대로 유지 (비정규화 단일 테이블)
- **Option B**: Star Schema (Fact + Dimension 테이블 분리)

---

## 의사결정

**Star Schema(Option B)를 선택합니다.**

### 근거

1. **쿼리 성능**: 대용량 테이블 전체 스캔 대신 Fact 테이블 중심의 효율적인 조인 가능
2. **대시보드 요구사항**: 준실시간 업데이트 주기를 고려할 때 빠른 쿼리 응답 속도 필수
3. **비즈니스 확장성**: 새로운 분석 요구사항 발생 시 Dimension 테이블만 수정하면 되어 유연한 구조 확보
4. **Surrogate Key**: 자연키 대신 Surrogate Key를 사용하여 중복 집계 문제 방지 (SCD Type 1 적용)

### 테이블 설계

```
Fact Table: fact_user_events (~1,280,000 rows)
  ├── dim_user       (~2,000 rows)   — 유저 정보, 세그먼트, A/B 그룹
  ├── dim_recipe     (~10,000 rows)  — 레시피 메타데이터
  ├── dim_event      (~15 rows)      — 이벤트 타입 정보
  ├── dim_page       (~20 rows)      — 페이지 정보
  └── dim_time       (~17,520 rows)  — 시간 차원 (년/월/일/시, 요일, 주말 여부)
```

### SCD Type 1 선택 근거

- 유저 속성(세그먼트, A/B 그룹)의 **현재 상태**가 분석에 더 중요
- 히스토리 추적보다 최신 상태 유지가 비즈니스 요구사항에 부합
- Type 2 대비 구현 복잡도와 스토리지 비용 절감

---

## 결과

- Raw Data 대비 쿼리 스캔량 **48.7% 감소** (Fact 테이블 기준)
- 스키마 크기 **45.6% 감소**
- Trino + Superset 연동으로 대용량 로그 데이터에서도 준실시간 대시보드 조회 가능

## 트레이드오프

| 항목 | Star Schema | 단일 비정규화 테이블 |
|------|------------|------------------|
| 쿼리 성능 | 우수 | 보통 (Full Scan 위험) |
| 구현 복잡도 | 높음 (조인 설계 필요) | 낮음 |
| 유지보수 | 용이 (Dimension 독립 수정) | 어려움 (스키마 변경 영향 큼) |
| 스토리지 | 효율적 | 중복 데이터 존재 |
| SCD 처리 | Type 1 적용 (히스토리 미보존) | 해당 없음 |
