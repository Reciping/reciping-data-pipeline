# ADR 001. AWS 매니지드 서비스 대신 EC2 Self-Hosted 선택

- **날짜**: 2025.08
- **상태**: 승인됨

---

## 배경

데이터 파이프라인 인프라를 구성할 때 두 가지 선택지가 있었습니다.

- **Option A**: AWS 매니지드 서비스 (MSK, EMR, Glue 등)
- **Option B**: EC2에 직접 설치 및 운영 (Self-Hosted)

AWS 매니지드 서비스는 운영 편의성이 높지만, 스타트업 환경에서 **고정비 부담**이 핵심 제약 조건이었습니다.
또한 클라우드 환경에서 데이터 엔지니어가 인프라 기술 호기심을 갖고 동작 원리를 직접 이해하는 것이 중요하다고 판단했습니다.

---

## 의사결정

**EC2 Self-Hosted 방식(Option B)을 선택합니다.**

### 근거

1. **비용 제어**: 매니지드 서비스 대비 고정비를 직접 통제할 수 있음
2. **워크로드 맞춤 최적화**: 각 컴포넌트의 특성에 맞춰 인스턴스 타입을 개별 선정 가능
3. **기술 이해도**: 블랙박스 없이 파이프라인 전체 동작 원리를 직접 파악 가능
4. **커스텀 메트릭**: 비즈니스 워크로드 특성에 맞는 커스텀 메트릭 추출 체계 구축 가능

### 인스턴스 타입 선정 근거

| 컴포넌트 | 인스턴스 | 선정 이유 |
|---------|---------|----------|
| Spark / Airflow | t3.xlarge | Spark OOM 방지를 위한 메모리 확보 |
| Trino | t3.large | 대화형 쿼리 처리에 최적화 |
| Kafka / Kafka Connect | t3.medium | 15분 간격 스트리밍 처리 및 인프라 균형 |
| Hive Metastore | t3.micro | 메타데이터 저장 전용, 경량 워크로드 |

---

## 결과

- AWS 매니지드 서비스 대비 인프라 고정비 **75% 절감** 달성
- 비즈니스 워크로드 특성에 최적화된 **커스텀 오토스케일링 + 메트릭 추출 체계** 구축

## 트레이드오프

| 항목 | Self-Hosted | 매니지드 서비스 |
|------|------------|---------------|
| 초기 설정 비용 | 높음 | 낮음 |
| 운영 부담 | 높음 (직접 관리) | 낮음 (AWS 관리) |
| 인프라 비용 | 낮음 | 높음 |
| 커스터마이징 | 자유로움 | 제한적 |
| 장애 대응 | 직접 대응 필요 | AWS SLA 보장 |
